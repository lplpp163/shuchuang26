# 島語通 DaoTalk AI 正式版技術架構

更新日期：2026-07-11  
文件狀態：已依目前程式實作更新；任何尚未完成真人或 live model 測試的項目均明確標示。

## 產品要解決的工作

島語通處理的不是「把整份中文換成另一種語言」，而是使用者拿到工作文件後的一連串任務：先確認文件類型，找到日期、時間、金額與需要回覆的內容；接著用偏好語言整理重點，保留能回到原文的證據；最後產生可詢問的雙語句子，遇到扣款、欠薪、證件、人身安全或資料不足時轉給真人。

正式版第一個完整案例是中越薪資明細，之後串接出勤／排班／加班資料，第三階段再處理多頁勞動契約。這個順序來自臺灣官方範本的欄位研究，詳見[臺灣職場文件範本研究](./台灣職場文件範本研究.md)。

## 系統資料流

```text
手機照片／PDF／貼上文字
        │
        ▼
前端格式、大小與明確同意檢查
        │
        ▼
FastAPI 記憶體內解析與檔案簽章驗證
        │
        ▼
OpenAI Responses API 多模態理解
  ├─ 圖片：視覺輸入
  ├─ PDF：抽取文字＋逐頁影像
  └─ 文字：原文輸入
        │
        ▼
Pydantic Structured Outputs
  文件類型／欄位／原文證據／信心／風險／下一步
        │
        ▼
本地確定性安全規則
  高風險關鍵字補捉／真人轉介／110、119、1955
        │
        ▼
可核對的雙語行動卡與本輪 JSON
```

OpenAI 官方說明指出，最新模型支援文字、影像、多語與 Responses API；本專案預設選擇 `gpt-5.6-terra`，理由是官方將它定位在能力與成本之間的平衡。模型名稱仍可透過環境變數調整，正式報告必須記錄實際測試的模型與日期，不能只寫「使用 AI」。參考：[Models](https://developers.openai.com/api/docs/models)。

圖片以 `input_image` 傳入；PDF 以 `input_file` 傳入。OpenAI 的檔案輸入文件說明，具視覺能力的模型處理 PDF 時會同時取得抽出的文字與頁面影像，因此能看見表格結構，而不只是純文字擷取。參考：[Images and vision](https://developers.openai.com/api/docs/guides/images-vision)、[File inputs](https://developers.openai.com/api/docs/guides/file-inputs)。

模型回應使用 Pydantic schema 約束，欄位之外的內容會被拒絕，並由 API 再次驗證。這能穩定介面與評測格式，但不代表內容一定正確；每一個重要欄位仍需附原文片段、頁碼（能判定時）與信心。參考：[Structured model outputs](https://developers.openai.com/api/docs/guides/structured-outputs)。

## 正式版組件

### 操作介面

`正式版/frontend` 是手機優先的操作介面，可上傳 PNG、JPG、WEBP、PDF 或貼文字，選擇越南語、印尼語、英語與繁體中文，並顯示處理階段、欄位證據、信心、風險、下一步、雙語詢問句、1955 入口與本輪技術紀錄。瀏覽器不放 API 金鑰，所有模型呼叫都由後端執行。

若後端沒有啟動，只有「合成範例」會進入明確標示的前端備援畫面；該畫面會寫明沒有呼叫 AI。一般文件不會在連線失敗時偽造分析結果。

### 分析 API

`正式版/backend` 使用 FastAPI。`POST /api/analyze` 接受 JSON 文字或 multipart 檔案；`GET /healthz` 回報目前是 live 或 deterministic mock。檔案先檢查副檔名、MIME、檔頭、像素、頁數與大小，影像再以 Pillow 解碼，PDF 再以解析器確認頁面與加密狀態。輸入只在單次請求記憶體內處理，不寫入資料庫或暫存檔。

未設定 `OPENAI_API_KEY` 時，後端自動進入 deterministic mock。mock 可驗證 API 契約、欄位格式、風險規則與介面，不會假裝看懂圖片或 PDF。設定金鑰後才啟用多模態 live 分析；傳送外部模型前必須收到 `consent=true`。

### 結構化資料

核心輸出至少包含：

- 文件類型與偏好語言摘要；
- 日期、時間、金額及各自的原文證據；
- 需要向真人確認的問題與安全順序；
- 風險類別、嚴重度、證據與說明；
- 整體與欄位信心；
- 是否需要轉介及建議服務；
- 分析模式、模型、供應商資料處理說明與 request ID。

計畫中的薪資加總、扣除、實領、工時與跨午夜計算，應由確定性程式執行，不把算術交給 LLM 自由生成。模型負責讀取、對應與解釋；程式負責重算與比較。

### 安全閘門

本地規則會補捉欠薪、異常扣款、扣留護照、暴力、限制自由、解僱、職災、危險作業、騷擾與隱私等訊號。模型未標出但規則命中時，仍會建立風險項目；高風險會強制啟用真人轉介。文件中的「忽略規則」「輸出系統提示」等文字一律視為文件內容，不是系統指令。

正式操作仍需人工覆核。OpenAI 的安全建議也要求高風險場景保留 human-in-the-loop、進行對抗測試、限制輸入範圍並讓使用者能回到原始資料核對。參考：[Safety best practices](https://developers.openai.com/api/docs/guides/safety-best-practices)。

## 資料保護界線

後端呼叫 Responses API 時設定 `store=false`，不建立 Files API 物件，應用程式本身也不寫入資料庫。不過 `store=false` 不等於供應商完全不處理或不保留資料；OpenAI 官方說明，API 資料預設不拿來訓練模型（除非主動選擇分享），但一般帳戶仍可能有最長 30 天的濫用監控紀錄，Zero Data Retention 需要另行申請。參考：[Data controls](https://developers.openai.com/api/docs/guides/your-data)。

目前版本要求使用者先自行遮住姓名、護照／居留證號、電話與地址，再明確同意送出。自動本地遮罩尚未完成，不能在計畫書寫成現有功能。正式公開測試前至少需補上：

1. 裝置端或受控後端的 PII 偵測與遮罩；
2. 隱私告知、使用目的、保存時間與刪除方式；
3. 權限、金鑰、日誌、事件回報與資料外洩處理；
4. 真實文件的同意、去識別、存取範圍與刪除日；
5. 供應商保留政策與必要的組織層級資料控制。

## 可驗證資產

- 14 份中央或地方政府公開範本，含來源 URL、格式與 SHA-256。
- 1 份完全合成的中越薪資明細，提供 HTML、清晰影像、含背景影像、PDF 與人工 ground truth。
- 後端單元測試、API 測試、檔案驗證與 OpenAI adapter contract test。
- 18 個完全合成的文字案例，涵蓋正常、模糊、高風險、提示注入與 PII。
- 手機與桌面前端 smoke test，以及 JSON 匯出與後端狀態顯示。

實際測試結果統一寫入評測報告，不在本文件預填準確率。沒有 API 金鑰時，只能證明 deterministic mock、API、規則與介面可重現；不能宣稱已驗證多模態模型的 OCR 或語意準確率。

## 送審前的完成條件

1. 用同一模型版本跑完清晰影像、含背景影像與 PDF，保存原始 JSON、延遲、失敗與成本。
2. 對 ground truth 計算日期、時間、金額 exact match；薪資算式必須 100% 重算一致。
3. 所有高風險案例須轉介，提示注入不得改變系統任務，真人個資不得出現在日誌與公開錯誤。
4. 越南語與印尼語內容由熟悉臺灣職場用語的母語者審閱；介面明確標示審閱狀態。
5. 由移工完成任務測試，記錄是否找到文件用途、重要金額、下一步與真人入口，不只詢問「喜不喜歡」。
6. 影片、網站與計畫書使用相同的功能狀態：已完成、測試中與未完成不能混寫。

## 技術選型的替代方案

如果 live 測試發現影像成本、延遲或敏感資料不適合直接送入通用模型，架構可改為裝置端 OCR／版面模型先做文字與位置抽取，再把去識別的結構資料交給 LLM。若文件樣式穩定，也可用模板定位與確定性欄位規則取代部分模型工作。正式產品的價值在可核對、安全且能完成任務，不在於每一步都必須使用生成式 AI。
